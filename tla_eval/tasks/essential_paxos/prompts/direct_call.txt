TASK
====

Write a TLA+ specification of single-decree Essential Paxos that faithfully
models the reference Python implementation provided below. The specification
will be evaluated for syntax correctness (SANY), runtime correctness (TLC),
conformance to captured execution traces of the reference implementation,
and satisfaction of expert-written safety invariants.


SCOPE — single-decree Paxos only
================================
- Model three roles: Proposer, Acceptor, Learner. Multiple of each may exist.
- Model two phases: (1) Prepare / Promise, (2) Accept / Accepted.
- Model exactly these message types: Prepare, Promise, Accept, Accepted.
- Acceptor state: promised proposal id, accepted proposal id, accepted value.
- Proposer state: current proposal id, proposed value, set of promises
  received, highest previously-accepted id learned via promises.
- Learner state: per-acceptor latest accepted proposal id, and once
  consensus is reached, the final decided value.
- Quorums are simple majorities, parameterized by the number of acceptors.

OUT OF SCOPE — do not include any of the following:
- Multi-Paxos, log replication, or sequence numbers across multiple
  consensus instances.
- Leader election, heartbeats, or NACKs.
- Durability, persistence, or crash recovery.
- Message retransmission (the network model handles this implicitly).
- Byzantine faults.
- Liveness properties or fairness assumptions.

Anything not listed in SCOPE must not appear in the specification.


NETWORK MODEL
=============
- Asynchronous messaging via a single global set `msgs` of in-flight
  messages.
- Sending a message is modeled as `msgs' = msgs \union {m}`.
- Receiving a message reads it from `msgs` without removing it, so the same
  message may be delivered to multiple recipients and may be re-read.
- This implicitly models loss, duplication, and reordering: any subset of
  sent messages may be processed in any order.
- Do not introduce per-link queues, sequence numbers, or acknowledgements.


OUTPUT FORMAT
=============
- A single TLA+ module named `EssentialPaxos`.
- EXTENDS Naturals, FiniteSets, TLC as needed.
- CONSTANTS: Acceptors, Proposers, Learners, Values (all sets), NoValue.
  (NoValue is a sentinel meaning "no value decided/accepted yet". Declare it
  as a CONSTANT — the .cfg will bind it to a model value. Never define it
  as `CHOOSE x : x \notin Values` — TLC cannot evaluate unbounded CHOOSE.)
- Define Quorums as the set of majority subsets of Acceptors.
- Add a CONSTANT MaxBallot and define ProposalIDs == (0..MaxBallot) \X Proposers.
  Using Nat \X Proposers makes the state space infinite and TLC will fail.
  The .cfg will bind MaxBallot to a small integer (e.g. 3).
- Because TLC cannot apply `>` or `<` to tuples directly, define an explicit
  comparison operator. CRITICAL: the second element of each ProposalID is a
  Proposer, which is a TLC model value. TLC model values do NOT support `<`
  comparison — `a[2] < b[2]` on model values will throw a TLC runtime error.
  Therefore, compare only on the numeric ballot component:
    ProposalIDLt(a, b) == a[1] < b[1]
  To keep this a valid total pre-order, ensure each proposer starts at a
  different ballot number in Prepare: proposer p uses a per-proposer counter
  so the same ballot number is never assigned by two different proposers
  simultaneously. Use this operator everywhere proposal IDs are compared.
  Never write `pid1 > pid2` on tuple-typed values, and never write `a[2] < b[2]`
  where a[2] is a model value from Proposers or Acceptors.
- Quorum checks: always check quorum membership by testing
    Cardinality(someSet) * 2 > Cardinality(Acceptors)
  Never use `CHOOSE n \in Nat : ...` for quorum size — Nat is infinite and
  TLC cannot evaluate such CHOOSE expressions. Compute the check inline.
- VARIABLES: declare each variable on its own line with a plain name (no
  record-valued grouping). Do NOT group variables into a single record-typed
  variable per role — declare each field as a separate VARIABLE. This avoids
  multi-line nested record type expressions that SANY cannot parse.
  Example (correct):
    VARIABLES proposerProposalId, proposerValue, ..., msgs
  Wrong (causes SANY parse error):
    VARIABLES proposerState  \* where proposerState is typed as [Proposers -> [...]]
- A TypeOK invariant covering every variable. In TypeOK, keep each type
  expression on one line, or if it must span lines, keep the closing `]]` on
  the same line as the last field. Never break a multi-line record type such
  that the closing bracket `]]` is on its own line at the same indentation
  level as the conjunction `/\` — SANY will reject it as a parse error.
- An Init predicate setting all variables to a clearly defined starting
  state.
- A Next predicate that is a disjunction of these named actions
  (existentially quantified over their arguments):
    Prepare(p)                 -- a proposer initiates Phase 1
    HandlePrepare(a, m)        -- an acceptor processes a Prepare message
    HandlePromise(p, m)        -- a proposer processes a Promise message
    HandleAccept(a, m)         -- an acceptor processes an Accept message
    HandleAccepted(l, m)       -- a learner processes an Accepted message
- A `vars` tuple listing every variable.
- Spec == Init /\ [][Next]_vars.
- Output only the TLA+ module, wrapped in a single ```tla code block. No
  prose explanation before or after.
