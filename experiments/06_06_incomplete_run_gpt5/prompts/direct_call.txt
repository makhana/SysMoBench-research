Essential Paxos modeling requirements:

MANDATORY CORE ACTIONS (must include all):
1. [Prepare] A proposer starts Phase 1, creates a fresh proposal id, resets its promise set, increments its next proposal number, and broadcasts Prepare messages to all acceptors.
2. [HandlePrepare] An acceptor processes a Prepare message. If the proposal id equals its current promise, it resends a Promise. If the proposal id is greater than its current promise, it records the new promise and sends a Promise. If the proposal id is lower, it ignores the message.
3. [HandlePromise] A proposer processes a Promise. It ignores promises for old proposal ids and duplicate promises from the same acceptor. Otherwise it records the promise, adopts the highest previously accepted non-empty value reported by acceptors, and broadcasts Accept messages once a majority has promised and the proposer has a value.
4. [HandleAccept] An acceptor processes an Accept request. If the proposal id is at least its promised id, it records the promise, accepted proposal id, and accepted value, then sends Accepted messages to all learners. If the proposal id is lower, it ignores the request.
5. [HandleAccepted] A learner records Accepted messages from acceptors, keeps only the latest proposal id seen from each acceptor, and decides a final proposal/value after a majority of acceptors report the same proposal id and value.

REQUIRED BEHAVIORAL SCOPE:
- Model three roles: Proposer, Acceptor, and Learner. Multiple instances of each role may exist.
- Model exactly these network message kinds: Prepare, Promise, Accept, Accepted.
- Use one global asynchronous network variable `msgs` containing in-flight messages.
- Sending a message is modeled by adding a record to `msgs`.
- Receiving a message reads a selected record from `msgs` without removing it. This intentionally permits loss, duplication, reordering, and repeated delivery.
- Message records should carry enough information to match the Python handlers:
  * Prepare: proposer sender, acceptor recipient, proposal id
  * Promise: acceptor sender, proposer recipient, proposal id, previously accepted proposal id, previously accepted value
  * Accept: proposer sender, acceptor recipient, proposal id, value
  * Accepted: acceptor sender, learner recipient, proposal id, value
- Acceptor state must include promised proposal id, accepted proposal id, and accepted value.
- Proposer state must include current proposal id, proposed value, next proposal number, received promises, and highest previously accepted proposal/value learned from promises.
- The Python `set_proposal(value)` input may be modeled either as finite initial proposer values or as an auxiliary `SetProposal(p, v)` action. If you include `SetProposal`, it may appear in `Next`, but it is not one of the five trace-validation actions and must not replace `Prepare`.
- Learner state must include the latest accepted proposal id per acceptor, enough per-proposal counts to decide by quorum, and the final decided proposal/value.
- Quorums are simple majorities of `Acceptors`.
- A learner decides only when a majority of acceptors report the same proposal id and value.
