You are a TLA+ expert. Generate one complete TLC configuration file (.cfg) for the Essential Paxos specification below.

## Input Specification

$tla_spec

## Generated Invariants

$invariants

## Requirements

1. Analyze the specification and identify the exact specification operator and every declared CONSTANT.
2. Use `SPECIFICATION Spec` when the module defines `Spec`; otherwise use the exact specification operator defined by the module. Do not use separate INIT/NEXT clauses unless no specification operator exists.
3. Assign every declared constant with concrete finite values. Do not bind constants that the spec does not declare.
4. Prefer this small but non-trivial Paxos model when the matching constants exist:
   - `Acceptors = {a1, a2, a3}`
   - `Proposers = {1, 2}`
   - `Learners = {l1, l2}` (or `{l1}` if the spec only supports one learner)
   - `Values = {v1, v2}`
   - `NoValue = NoValue`
   - `MaxBallot = 4`
5. Use bare TLC model values for acceptors, learners, values, and `NoValue`; do not quote them as strings.
6. Do not bind `NoProposal` unless the specification declares it as a CONSTANT. Preferred Essential Paxos specs define `NoProposal == <<0, 0>>` inside the module, because TLC cfg files in this pipeline cannot bind tuple expressions such as `<<0, 0>>` to constants.
7. Do not bind a `ProposerRank` constant unless the specification declares it. The preferred Essential Paxos prompt no longer uses it; proposer ids are numeric instead.
8. If the spec uses `BallotNumbers` or another finite ballot constant instead of `MaxBallot`, bind it to a small finite set such as `{1, 2, 3, 4}`.
9. If the spec declares extra finite constants, assign the smallest sensible values that keep Prepare/Promise/Accept/Accepted behavior possible.
10. Include `CHECK_DEADLOCK FALSE` because single-decree Paxos can legitimately stop after a decision.
11. Include `INVARIANT TypeOK` only if `TypeOK` is actually defined in the specification.
12. Include provided generated invariants only if they are present in the Generated Invariants section or already defined in the specification text. State predicates go under `INVARIANT`; temporal formulas using `[]`, `<>`, `~>`, or primed variables go under `PROPERTY`.
13. Do not invent expert invariants such as `Agreement`, `Validity`, `Stability`, or `PromiseMonotonic` when they are not defined or provided.
14. Do not declare `SYMMETRY`; it can obscure trace-validation and invariant debugging for this task.
15. Output only raw .cfg content: no markdown, no code fences, and no explanations.
